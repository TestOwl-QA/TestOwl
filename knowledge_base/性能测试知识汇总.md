# 性能测试知识汇总（移动设备重点）

## 一、概述

性能测试是确保游戏和应用程序在各种设备、环境下流畅运行的关键环节。移动设备由于其硬件限制（CPU性能通常仅为桌面设备的1/4、内存容量4-8GB）、网络波动（移动网络延迟300-1000ms）和使用场景特殊性，性能测试尤为重要。

### 性能劣化的商业影响
- **用户流失**：帧率波动超过30%将导致53%玩家放弃游戏
- **设备兼容性**：不同硬件配置下性能表现差异显著
- **运营成本**：服务器性能不足增加带宽与维护开销

## 二、核心性能指标

### 2.1 帧率（FPS）

| 场景类型 | 目标帧率 | 说明 |
|---------|---------|------|
| 竞技类游戏 | 144FPS+ | 职业选手需要240FPS |
| 动作游戏 | 稳定60FPS | 波动不超过10% |
| 策略游戏 | 30FPS可接受 | UI响应必须流畅 |
| VR游戏 | 90FPS | 防止眩晕的最低要求 |

**帧时间（Frame Time）**：更准确反映性能波动
- P99帧时间：99%的帧都能在此时间内完成
- 卡顿阈值：>33.3ms（30 FPS）的帧

**体验质量公式**：
```
QoE = α × FPS_avg - β × σ_FPS - γ × N_stutter
```
其中β（波动惩罚）的权重往往大于α（平均值奖励）。

### 2.2 CPU/GPU占用

| 指标 | 正常范围 | 异常阈值 |
|-----|---------|---------|
| CPU占用率（空闲状态） | 0% | - |
| CPU占用率（中等规格） | ≤30% | >50% |
| CPU占用率（正常运行） | ≤50% | >80% |
| GPU占用率 | 依场景而定 | 持续95%+ |

**CPU高占用原因**：代码逻辑过多、AI/物理/寻路消耗
**GPU高占用原因**：渲染、粒子、后处理、动态灯光

### 2.3 内存占用

| 测试状态 | 关注点 | 异常标准 |
|---------|-------|---------|
| 空闲状态 | 最小内存消耗 | - |
| 中强度状态 | 是否卡顿 | - |
| 强度状态 | 内存泄漏 | 持续增长 |
| 内存泄漏检测 | 资源释放 | 增量>50MB/操作 |

**内存泄漏标准**：内存泄漏<2MB/小时

### 2.4 启动时间

| 启动类型 | 标准要求 | 测试方法 |
|---------|---------|---------|
| 冷启动 | ≤1.5s | FFmpeg录屏拆帧 / adb命令 |
| 热启动 | ≤1s | adb shell am start -W |

### 2.5 加载时长

- 场景加载：≤5秒
- 首帧加载：≤3秒
- 关卡切换：≤2秒

### 2.6 发热与耗电

| 测试场景 | 正常范围 | 异常阈值 |
|---------|---------|---------|
| 待机电流 | ≤2mA | >5mA |
| 游戏发热 | 温升<10℃ | 手持烫手 |
| 电池消耗 | 依游戏类型 | 异常快速掉电 |

### 2.7 网络延迟

| 场景 | 目标延迟 | 可接受延迟 |
|-----|---------|----------|
| 即时对战 | <50ms | <100ms |
| MOBA类 | <80ms | <150ms |
| MMORPG | <100ms | <200ms |

### 2.8 GPU过度渲染

| 绘制级别 | 颜色标识 | 优化建议 |
|---------|---------|---------|
| 无过度绘制 | 原色 | - |
| 1x | 蓝色 | - |
| 2x | 绿色 | - |
| 3x | 浅红 | 可优化 |
| 4x | 深红 | 必须优化 |

**测试指标**：
- 控制过度绘制≤2x
- 不允许存在4x过度绘制
- 不允许面积超过屏幕1/4的3x过度绘制

### 2.9 流量消耗

| 测试场景 | 测试内容 |
|---------|---------|
| 冷启动 | 首次启动到全部加载完成耗流 |
| 热启动 | 非首次启动到全部加载完成耗流 |
| 后台运行 | 后台运行耗流 |
| 特定场景 | 运行某场景消耗的总流量 |

## 三、移动设备性能测试方法

### 3.1 测试设备选择

**Android设备选择策略**：
- 4G内存设备覆盖用户最多（37.6%）
- 6G以上内存用户占比50.1%
- 3G用户8.7%，2G以下2.6%

**测试设备分级**：
| 分级 | 设备特征 | 测试重点 |
|-----|---------|---------|
| 低端机 | 2-3G内存，老旧CPU | 基本可玩性、闪退测试 |
| 中端机 | 4-6G内存，主流CPU | 性能基准、流畅度 |
| 高端机 | 8G+内存，旗舰CPU | 极限表现、画质测试 |

### 3.2 FPS测试方法

**方法一：adb命令**
```bash
# 打开开发者选项 -> GPU呈现模式分析
adb shell dumpsys gfxinfo 包名

# 输出解读
# Draw: Java创建显示列表时间（OnDraw方法）
# Process: 渲染引擎执行显示列表时间
# Execute: 帧数据发送到屏幕时间
# Draw + Process + Execute < 16ms 保证60FPS
# Janky frames: 丢帧率
```

**方法二：开发者选项图形显示**
```
开发者选项 -> GPU呈现模式分析 -> 在屏幕上显示为条形图
绿线=16ms分隔线，直接观察流畅度
```

**方法三：第三方工具**
- PerfDog（推荐）
- Emmagee
- GT
- Android Studio Monitor

### 3.3 CPU测试方法

**方法一：adb命令**
```bash
# 查看某软件CPU占用率
adb shell top -m cpu | grep packageName

# 查看CPU占用前10的应用
adb shell top -m 10 -s cpu

# 一段时间的平均值
adb shell dumpsys cpuinfo | grep 包名
```

**方法二：第三方工具**
- PerfDog（推荐）
- Emmagee
- GT
- Android Studio Monitor

### 3.4 内存测试方法

**方法一：adb命令**
```bash
# 获取内存信息
adb shell dumpsys meminfo packageName

# 关注参数
# native heap alloc: JNI层内存分配
# dalvik heap alloc: Java层内存分配
# pss: 应用真正占据的内存大小

# 注意：如果前两个值一直增长，可能内存泄漏
```

**方法二：第三方工具**
- PerfDog（推荐）
- Emmagee
- Android Monitor
- DDMS Heap

### 3.5 GPU过度渲染测试

```bash
# 开发者选项 -> 调试GPU过度绘制 -> 显示过度绘制区域
# 原色：无过度绘制
# 蓝色：绘制一次
# 绿色：绘制两次
# 浅红：绘制三次（可优化）
# 深红：绘制四次（必须优化）
```

### 3.6 电量测试方法

**方法一：第三方工具**
- Emmagee
- GT
- 待机电流对比≤2mA

**方法二：adb命令**
```bash
# 获取电量消耗信息
adb shell dumpsys batterystats | grep packageName

# 切换电池为非充电状态
adb shell dumpsys battery set status 1

# 获取整个设备电量消耗
adb shell dumpsys batterystats | more

# 获取某个APK电量消耗
adb shell dumpsys batterystats com.Package.name | more
```

### 3.7 流量测试方法

**方法一：第三方工具**
- Emmagee
- GT
- PerfDog

**方法二：系统TCP计数**
```bash
# 查找app对应Uid
adb shell dumpsys package packageName | grep userId=
adb shell ps | grep packageName
adb shell cat /proc/{pid}/status | grep Uid

# 获取下行流量
adb shell cat /proc/uid_stat/{Uid}/tcp_rcv

# 获取上行流量
adb shell cat /proc/uid_stat/{Uid}/tcp_sd
```

**方法三：抓包工具**
- Charles
- Fiddler
- 可定位具体接口流量异常

## 四、性能测试流程

### 4.1 测试准备

1. **明确测试目标**
   - 测什么：帧率、内存、CPU、GPU、加载时长、发热、网络延迟
   - 测哪些平台：安卓、iOS、PC、主机、模拟器
   - 测什么场景：主界面、战斗、多人联机、开放世界、资源加载

2. **准备设备和工具**
   - 低中高端主流机型各备一个
   - 旧机型、满存、各种网络环境
   - 工具：Unity/Unreal Profiler、PerfDog、FPS计数器

3. **建立测试标准**
   - 不同平台/机型目标值分档
   - 写成文档，推动团队认同

### 4.2 测试执行流程

```
1. 测试环境搭建
   ↓
2. 构建测试用例
   ↓
3. 安装测试工具
   ↓
4. 执行基础性能测试
   ↓
5. 数据采集与分析
   ↓
6. 问题整理与报告输出
   ↓
7. 性能优化回溯与复测
   ↓
8. 持续集成与版本跟踪
```

### 4.3 测试场景设计

| 测试类型 | 测试内容 | 目的 |
|---------|---------|-----|
| 基准测试 | 标准化场景性能评估 | 获取性能基线 |
| 压力测试 | 极限负载下表现 | 发现性能瓶颈 |
| 稳定性测试 | 长时间运行（8-12小时） | 检测内存泄漏 |
| 兼容性测试 | 多设备/系统版本 | 适配验证 |
| 网络测试 | 弱网/断网/切换 | 网络容错性 |

## 六、性能优化方法

### 6.1 资源优化

| 资源类型 | 优化方法 |
|---------|---------|
| 纹理 | 使用ASTC/ETC压缩格式，尺寸≤1024，使用Mipmap |
| 模型 | 控制面片数量（动态300-2000，静态<500顶点） |
| 动态物体 | 控制骨骼<30根，材质1-3个，Skinned Mesh Render=1 |
| 音频 | 背景音乐用OGG/MP3，短音效用WAV/AIF |
| 地形 | 高度图<257，混合纹理≤4个 |

### 6.2 渲染优化

| 优化技术 | 说明 |
|---------|-----|
| DrawCall Batching | 合并物体减少DrawCall |
| Static Batching | 静态物体合并，静态VB+动态IB |
| Dynamic Batching | 动态物体合并，支持<900顶点 |
| GPU Instancing | 相同Mesh和材质批量渲染 |
| LOD | 远近不同精度模型 |
| 遮挡剔除 | 不渲染被遮挡物体 |
| 视锥剔除 | 不渲染视野外物体 |

### 6.3 CPU优化

| 优化方向 | 具体方法 |
|---------|---------|
| 对象创建 | 轻量对象代替重量对象，后台线程创建，缓存池复用 |
| 布局计算 | 后台线程提前计算，缓存布局 |
| 代码执行 | 分帧执行，减少Update中临时变量，定时GC |
| 数学运算 | 用int代替float，用乘法代替除法 |
| 并发控制 | 控制线程最大并发数，耗时操作放子线程 |

### 6.4 内存优化

| 优化方向 | 具体方法 |
|---------|---------|
| 对象池 | 复用高频创建销毁对象 |
| 资源管理 | 及时释放不再使用的资源 |
| 字符串 | 使用StringBuilder拼接 |
| 数据结构 | 使用struct代替class（简单数据） |
| 场景切换 | 彻底资源回收 |

### 6.5 发热优化

| 原因 | 优化方法 |
|-----|---------|
| CPU/GPU高负载 | 降低并发量，分帧处理 |
| 特效密集 | 控制粒子、特效密度 |
| 持续运算 | 降帧率，自适应性能 |
| 多线程密集 | 优化AI/物理运算 |

## 七、常见性能问题与解决方案

### 7.1 帧率不稳定（掉帧）

**原因**：
- 动态物体/粒子过多
- 渲染批次复杂，DrawCall高
- Shader复杂
- 特效全开压垮GPU
- AI逻辑没分帧处理

**解决方案**：
- 降低批量实例数量，用对象池重用
- Shader简化，批量处理粒子/特效合并
- 场景远景粗处理，近景精细显示
- AI逻辑分步执行

### 7.2 内存占用爆炸（闪退）

**原因**：
- 不断申请新资源但未释放
- UI、对象池泄漏
- 视频、音频、图片资源没回收
- 多场景切换没清理

**解决方案**：
- 主动管理资源生命周期
- 内存快照工具查找泄漏根因
- UI界面复用或局部重建
- 场景切换时彻底资源回收

### 7.3 发热烫手机

**原因**：
- CPU/GPU激烈、任务堆积
- 持续高负载特效、AI循环

**解决方案**：
- 降低并发量，分帧处理
- 控制粒子、特效密度
- 降帧率，自适应性能
- 持续采集发热，识别高峰自动降负载

### 7.4 网络延迟高、掉线

**原因**：
- 数据包太大、同步频率高
- 服务器响应慢
- 客户端断网无重试机制

**解决方案**：
- 数据包瘦身，只传必要信息
- 降低同步频率
- 异步重试，断网后自动同步

## 八、性能测试报告结构

### 8.1 报告框架

```
1. 概述：本轮测试的目标与范围
2. 测试环境：设备、系统版本、网络环境
3. 测试方法：工具、核心流程、用例描述
4. 测试数据：帧率、内存、CPU/GPU、发热、耗电等表格和曲线图
5. 主要问题点：按优先级汇总，附截图/数据定位
6. 优化建议：针对每个大问题给出解决思路
7. 结论：哪些指标达标、哪些不达标，是否符合上线要求
```

### 8.2 优先级排序

| 优先级 | 问题类型 | 处理时效 |
|-------|---------|---------|
| P0 | 闪退、卡死、首帧读条>10秒 | 立即处理 |
| P1 | 频繁掉帧、发热严重 | 当日处理 |
| P2 | 偶发卡顿、局部掉帧 | 版本内跟进 |
| P3 | 轻微性能问题 | 持续优化 |

## 九、性能测试常见误区

| 误区 | 正确做法 |
|-----|---------|
| 只测旗舰机型，忽略低端设备 | 低端机测试更重要，覆盖市场大多数 |
| 测试流程只做一次 | 性能测试是持续工程，每个版本都测 |
| 只测主流程场景 | 异常场景、长时间游玩才是易崩点 |
| 只看主观感受 | 自动化采集数据最靠谱 |
| 发现问题只做表面修补 | 深挖瓶颈，追根溯源 |
| 不重视多平台兼容性 | 不同平台硬件差异大，都要覆盖 |

## 十一、术语表

| 术语 | 全称 | 说明 |
|-----|-----|-----|
| FPS | Frames Per Second | 每秒帧数，画面流畅度指标 |
| LCP | Largest Contentful Paint | 最大内容绘制时间 |
| FID | First Input Delay | 首次输入延迟 |
| CLS | Cumulative Layout Shift | 累积布局偏移 |
| PSS | Proportional Set Size | 比例分配共享内存后的私有内存 |
| DrawCall | - | 绘制调用，CPU通知GPU渲染的次数 |
| LOD | Level of Detail | 细节层次，根据距离调整模型精度 |
| GC | Garbage Collection | 垃圾回收，自动内存管理机制 |
| ANR | Application Not Responding | 应用无响应 |
| Overdraw | - | 过度绘制，同一像素多次绘制 |
