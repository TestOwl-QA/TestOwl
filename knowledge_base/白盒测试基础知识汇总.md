## 一、白盒测试概述

白盒测试（White Box Testing）是测试者了解代码结构，可以针对性地测试特定逻辑路径的测试方法。

### 1.1 白盒测试特点

| 特点 | 说明 |
|-----|------|
| 需要编程知识 | 测试人员需理解代码逻辑和数据结构 |
| 代码覆盖率分析 | 确保关键路径都被测试到 |
| 深度验证 | 验证伤害公式、概率系统等核心机制 |

### 1.2 典型应用场景

- 验证随机数生成器的分布均匀性
- 检查内存管理和资源释放
- 分析算法复杂度和性能瓶颈
- 确认异常处理和边界条件

## 二、逻辑覆盖测试方法

### 2.1 覆盖标准层级

```
语句覆盖 → 判定覆盖 → 条件覆盖 → 判定-条件覆盖 → 条件组合覆盖 → 路径覆盖
 (最弱)                                                      (最强)
```

### 2.2 语句覆盖（Statement Coverage）

**目标**：确保程序中的每一条可执行语句至少被执行一次

**示例代码**：
```python
a = 5
b = 3
if a > b:
    c = a - b
else:
    c = a + b
print(c)
```

**测试用例**：输入 a = 5, b = 3

**局限性**：
- 仅关注语句是否执行，无法发现逻辑判断错误
- 若将 `if a > b` 误写成 `if a < b`，该测试用例依然能通过

### 2.3 判定覆盖（Decision Coverage）

**目标**：每个判断的取真分支和取假分支至少被执行一次

**测试用例**：
- a = 5, b = 3（测试 a > b 的情况）
- a = 3, b = 5（测试 a <= b 的情况）

**优势**：比语句覆盖更严格，能发现判断条件相关错误

### 2.4 条件覆盖（Condition Coverage）

**目标**：每个判断中每个条件的可能取值至少满足一次

**示例代码**：
```python
if a > 0 and b < 10:
    # do something
```

**测试用例**：
- a = 1, b = 5（a > 0 为真，b < 10 为真）
- a = -1, b = 15（a > 0 为假，b < 10 为假）

**局限性**：可能忽略条件之间的组合关系

### 2.5 判定-条件覆盖（Decision-Condition Coverage）

**目标**：同时满足判定覆盖和条件覆盖的要求

**要求**：
1. 每个判断的取真/取假分支至少执行一次
2. 每个判断中每个条件的可能取值至少满足一次

### 2.6 条件组合覆盖（Condition Combination Coverage）

**目标**：每个判断中条件的各种可能组合都至少出现一次

**示例**：`if a > 0 and b < 10`

| 组合 | a > 0 | b < 10 | 测试数据 |
|-----|-------|--------|---------|
| 1 | 真 | 真 | a=1, b=5 |
| 2 | 真 | 假 | a=1, b=15 |
| 3 | 假 | 真 | a=-1, b=5 |
| 4 | 假 | 假 | a=-1, b=15 |

**注意**：测试用例数量随条件数量呈指数级增长

### 2.7 路径覆盖（Path Coverage）

**目标**：覆盖程序中所有可能的路径

**挑战**：复杂程序中路径数量众多，完全实现难度大

**策略**：尽可能覆盖主要路径，发现深层次逻辑错误

## 三、基本路径测试

### 3.1 测试流程

```
┌─────────────────────────────────────┐
│ 1. 将程序代码转换为控制流图           │
│    - 节点：语句或语句块               │
│    - 边：控制流走向                   │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 2. 计算控制流图的环形复杂度           │
│    - 决定独立路径数量                 │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 3. 根据独立路径设计测试用例           │
│    - 确保每条独立路径被测试到         │
└─────────────────────────────────────┘
```

### 3.2 环形复杂度计算

**公式**：`V(G) = E - N + 2`

- E：控制流图中的边数
- N：控制流图中的节点数
- V(G)：环形复杂度（独立路径数量）

## 四、代码审查方法

### 4.1 审查内容

| 检查项 | 说明 |
|-------|------|
| 语法正确性 | 代码是否能编译通过 |
| 规范性 | 命名、格式是否符合编码规范 |
| 可读性 | 代码是否易于理解 |
| 可维护性 | 代码是否易于修改和扩展 |
| 设计符合性 | 是否符合设计要求和编码标准 |

### 4.2 常见问题发现

- 是否有适当的注释
- 变量和函数命名是否合理
- 代码结构是否清晰
- 过度复杂的算法
- 资源泄漏风险
- 潜在的性能问题

---

## 五、静态分析

### 5.1 分析内容

- 语法错误
- 语义错误
- 代码结构问题
- 潜在的安全漏洞

### 5.2 常见检测项

| 问题类型 | 示例 |
|---------|------|
| 未使用的变量 | 定义后从未使用的变量 |
| 空指针引用 | 可能访问空指针的代码 |
| 未关闭的资源 | 文件句柄、数据库连接未关闭 |
| 代码复杂度 | 圈复杂度过高的函数 |

### 5.3 优势

- 开发早期发现问题
- 降低修复成本
- 提高开发效率

## 七、代码覆盖率指标

### 7.1 覆盖率类型

| 覆盖率类型 | 计算方式 | 目标值 |
|-----------|---------|--------|
| 语句覆盖率 | 已执行语句数 / 总语句数 | ≥80% |
| 分支覆盖率 | 已执行分支数 / 总分支数 | ≥70% |
| 函数覆盖率 | 已调用函数数 / 总函数数 | ≥90% |
| 行覆盖率 | 已执行代码行数 / 总代码行数 | ≥80% |

### 7.2 游戏开发中的覆盖率要求

| 模块类型 | 推荐覆盖率 | 说明 |
|---------|-----------|------|
| 核心战斗逻辑 | ≥90% | 关键功能，必须高覆盖 |
| 网络通信 | ≥85% | 稳定性关键 |
| UI逻辑 | ≥60% | 可适当降低 |
| 工具脚本 | ≥50% | 内部使用，可进一步降低 |
