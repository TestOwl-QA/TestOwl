## 一、编码基本原则

### 1.1 KISS 原则

**Keep It Simple, Stupid** - 简洁优先

- 避免使用生僻语法增加理解成本
- 减少"重复造轮子"，优先复用成熟工具
- 代码应直观易懂，减少歧义

### 1.2 YAGNI 原则

**You Aren't Gonna Need It** - 按需设计

- 基于当前明确需求设计方案
- 减少提前设计和过度设计
- 简单预测后续可能新增的功能，保留合理扩展性

### 1.3 渐进式开发

- 代码编写并非"一劳永逸"
- 正视需求的动态变化
- 设计时预留合理扩展性

### 1.4 统一样式

- 团队内部维持代码样式统一
- 统一样式比纠结"是否为行业主流样式"更重要
- 降低协作成本，提升代码可读性

## 二、命名规范

### 2.1 C# / Unity 命名规范

| 元素类型 | 命名风格 | 示例 |
|---------|---------|------|
| 常量 | 全大写+下划线 | `MAX_COUNT` |
| 类/接口/结构体/枚举 | 帕斯卡命名法 | `PlayerStats`, `IDamageable` |
| 接口 | I前缀+帕斯卡 | `IDamageable` |
| 私有字段 | 下划线+驼峰 | `_elapsedTimeInDays` |
| 公共属性/方法/事件 | 帕斯卡命名法 | `MaxHealth`, `DoorOpened` |
| 局部变量 | 驼峰命名法 | `playerName` |
| 序列化字段 | 驼峰命名法 | `health` |

### 2.2 Python 命名规范（PEP 8）

| 元素类型 | 命名风格 | 示例 |
|---------|---------|------|
| 模块/包 | 全小写+下划线 | `my_module` |
| 类 | 帕斯卡命名法 | `MyClass` |
| 函数/方法 | 小写+下划线 | `my_function` |
| 变量 | 小写+下划线 | `my_variable` |
| 常量 | 全大写+下划线 | `MAX_OVERFLOW` |
| 私有属性 | 单下划线前缀 | `_private_attr` |
| 名称重整 | 双下划线前缀 | `__mangled_name` |

### 2.3 应避免的名称

- **切勿**使用字符 'l'（小写L）、'O'（大写O）、'I'（大写I）作为单字符变量名
- 在某些字体中，这些字符与数字1和0无法区分
- 当想使用 'l' 时，改用 'L'

## 三、代码格式规范

### 3.1 缩进

- 使用**4个空格**进行缩进（不使用Tab）
- 保持项目内缩进风格一致

### 3.2 行长度

- 建议每行不超过**79-100个字符**
- 过长的行应适当换行

### 3.3 空行使用

- 类定义之间空**2行**
- 方法定义之间空**1行**
- 逻辑段落之间空**1行**

### 3.4 空格使用

```python
# 正确
x = 1
y = x + 1
if x == 1:
    pass

# 错误
x=1
y=x+1
if x==1:
    pass
```

## 四、注释规范

### 4.1 块注释

- 适用于其后的一些（或所有）代码
- 与代码缩进到同一级别
- 每行以 `#` 和一个空格开头
- 段落用包含单个 `#` 的行分隔

```python
# 这是块注释的第一段
# 可以有多行
#
# 这是第二段，用空行分隔
```

### 4.2 行内注释

- 与语句在同一行
- 与语句至少相隔两个空格
- 以 `#` 和一个空格开头
- 少量使用，避免显而易见的注释

```python
# 正确
x = x + 1  # Compensate for border

# 错误（显而易见）
x = x + 1  # Increment x
```

### 4.3 文档字符串（Docstring）

**Python 规范**：
```python
# 多行文档字符串
def function():
    """Return a foobang.
    
    Optional plotz says to frobnicate the bizbaz first.
    """

# 单行文档字符串
def function():
    """Return an ex-parrot."""
```

**C# 规范**：
```csharp
/// <summary>
/// 计算两个数的和
/// </summary>
/// <param name="a">第一个数</param>
/// <param name="b">第二个数</param>
/// <returns>两数之和</returns>
public int Add(int a, int b)
{
    return a + b;
}
```

## 五、事件规范（C#/Unity）

### 5.1 事件声明

优先使用 `event Action<T>` 或 `event Func<T>` 泛型委托

```csharp
// 推荐
public event Action OpeningDoor;
public event Action<int> PointsScored;
public event Action<CustomEventArgs> ThingHappened;

// 不推荐（自定义委托）
public delegate void MyEventHandler(int value);
public event MyEventHandler OnMyEvent;
```

### 5.2 事件触发方法

命名格式：`Raise` + 事件名

```csharp
public void RaiseDoorOpened()
{
    DoorOpened?.Invoke();  // 空值检查
}

public void RaisePointsScored(int points)
{
    PointsScored?.Invoke(points);
}
```

### 5.3 事件订阅方法

命名格式：订阅者类名_事件触发方法名

```csharp
public void PlayerController_OnDoorOpened()
{
    Debug.Log("门已打开");
}
```

### 5.4 多参数处理

当事件需要传递多个参数时，优先用 struct 结构体整合

```csharp
public struct CustomEventArgs
{
    public int ObjectId { get; }
    public Color Color { get; }
    
    public CustomEventArgs(int objectId, Color color)
    {
        ObjectId = objectId;
        Color = color;
    }
}
```

## 六、特性编写规范（C#/Unity）

**单个特性独占一行**，禁止多个特性挤在同一行

```csharp
// 正确
[Tooltip("单个特性独占一行")]
[SerializeField]
private float anotherStat;

// 错误
[Tooltip("多个特性同行")][SerializeField]
private float anotherStat;
```

## 七、序列化规范（C#/Unity）

### 7.1 访问权限

- 默认使用私有字段（private）配合 `[SerializeField]` 实现序列化
- **禁止**直接使用公共字段（public）序列化

### 7.2 访问控制

外部需访问的序列化字段，通过公共属性或方法暴露

```csharp
// 推荐
[SerializeField]
private int _maxHealth;

public int MaxHealth => _maxHealth;

// 不推荐
public int maxHealth;  // 直接公共字段
```

### 7.3 序列化范围

仅对需要在 Inspector 面板编辑或保存的字段添加 `[SerializeField]`

## 八、编程建议

### 8.1 比较

- 与 `None` 等单例对象比较应使用 `is` 或 `is not`
- 不要使用 `== True` 或 `== False` 比较布尔值

```python
# 正确
if foo is not None:
    pass
if greeting:
    pass

# 错误
if foo != None:
    pass
if greeting == True:
    pass
```

### 8.2 异常处理

- 捕获特定异常，而非裸 `except:`
- 使用 `raise X from Y` 表示异常链

```python
# 正确
try:
    value = collection[key]
except KeyError:
    return key_not_found(key)

# 错误
try:
    value = collection[key]
except:  # 捕获所有异常，不推荐
    return key_not_found(key)
```

### 8.3 资源管理

使用 `with` 语句确保资源及时清理

```python
# 正确
with open('file.txt', 'r') as f:
    content = f.read()

# 不推荐
f = open('file.txt', 'r')
content = f.read()
f.close()  # 可能忘记关闭
```

## 九、代码审查检查清单

### 9.1 命名检查

- [ ] 命名是否清晰表达意图
- [ ] 命名是否符合项目规范
- [ ] 是否避免了歧义名称
- [ ] 常量是否使用全大写

### 9.2 代码结构检查

- [ ] 函数长度是否合理（建议不超过50行）
- [ ] 类的职责是否单一
- [ ] 是否有重复代码（DRY原则）
- [ ] 嵌套层级是否过深（建议不超过3层）

### 9.3 注释检查

- [ ] 复杂逻辑是否有注释
- [ ] 注释是否准确描述代码行为
- [ ] 是否有过时或误导性注释
- [ ] 公共API是否有文档注释

### 9.4 错误处理检查

- [ ] 是否处理了所有可能的错误
- [ ] 错误信息是否清晰有用
- [ ] 资源是否正确释放
- [ ] 是否有适当的日志记录

## 十、团队协作配置

### 10.1 .editorconfig 文件

统一不同IDE的代码样式：

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true

[*.py]
indent_style = space
indent_size = 4

[*.cs]
indent_style = space
indent_size = 4
```

### 10.2 使用流程

1. 在 Rider/VS Code 中配置好代码样式
2. 导出 `.editorconfig` 文件
3. 将文件提交到项目仓库根目录
4. IDE 会自动识别并应用配置
